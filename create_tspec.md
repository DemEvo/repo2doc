# Спецификация на разработку утилиты RepoToDoc.

---

### 1. Ограничительная рамка (Core Restrictions & Boundary Conditions)
Любая спецификация начинается не с того, что нужно сделать, а с того, что делать категорически запрещено. Это сужает дерево поиска для исполнителя.

* ❌ **STRICTLY FORBIDDEN:** * Загрузка всего исходного кода репозитория в оперативную память (RAM) одновременно.
    * Использование внешних веб-сервисов или сторонних API для парсинга и обработки кода.
    * Попытка самостоятельного написания сложного парсера для `.gitignore` на стандартных библиотеках (во избежание пропуска мусора).
* ✅ **EXPLICITLY ALLOWED:** * Язык: строго Python 3.10+.
    * Использование сторонних библиотек `pathspec` (для точного парсинга `.gitignore`) и `GitPython` (для клонирования). Утилита должна поставляться с `requirements.txt`.
    * Чтение контента файлов — исключительно потоковым методом.

---

### 2. Детерминированный поток (State Machine / User Flow)
Весь процесс разложен на тактовые переходы состояний (State Machine). Каждое действие имеет субъекта.

1. `[Пользователь вызывает скрипт в CLI с аргументами]` -> `[Парсер аргументов валидирует флаги и пути]` -> `[Состояние: Валидация пройдена / Ошибка]`.
2. `[Состояние: Валидация пройдена (выбран --url)]` -> `[Система клонирует репозиторий через GitPython во временную директорию]` -> `[Состояние: Исходный код доступен локально]`.
3. `[Состояние: Исходный код доступен локально]` -> `[Pass 1: Система обходит директорию, применяет фильтры имен, расширений и pathspec для .gitignore. Список путей сохраняется в памяти]` -> `[Состояние: Отфильтрованный список путей сформирован]`.
4. `[Состояние: Отфильтрованный список путей сформирован]` -> `[Генератор дерева формирует строковое представление дерева, Писатель потока создает файл {output}_part1.txt и записывает стартовый блок]` -> `[Состояние: Файл готов к записи кода]`.
5. `[Система открывает следующий файл из списка (Pass 2)]` -> `[Парсер оценивает размер файла относительно --max-size]` -> `[Состояние: Файл одобрен / Файл превышает лимит]`.
6. `[Состояние: Файл одобрен]` -> `[Писатель потока построчно переносит текст файла в документ, обрабатывая возможные ошибки кодировки]` -> `[Состояние: Обновлен счетчик слов]`.
7. `[Счетчик слов превышает значение --split-words]` -> `[Система закрывает текущий текстовый документ, создает {output}_part{N+1}.txt и записывает "умный заголовок"]` -> `[Состояние: Запись переведена в новый файл]`.
8. `[Все файлы из списка обработаны]` -> `[Система удаляет временную директорию (если был --url)]` -> `[Состояние: Ресурсы очищены]`.
9. `[Состояние: Ресурсы очищены]` -> `[Система выводит итоговый консольный отчет]` -> `[Состояние: Процесс завершен]`.

### 3. Трансформация данных (Data Mutators & I/O)
Система должна понимать, как выглядят данные на входе и на выходе.

* **Входной формат (CLI-аргументы):** 
    * `--url https://github.com/user/repo` (Строка)
    * `--path ./local/repo` (Строка)
    * `--output repo_doc.txt` (Строка, default: `repo_doc.txt`)
    * `--extensions .py,.js,.html` (Строка со значениями через запятую)
    * `--max-size 200` (Integer, значение в КБ, default: `200`)
    * `--split-words 500000` (Integer, кол-во слов, default: `500000`)
* **Выходной формат (структура .txt документа):** 
    * *Блок 1: Сводка (только в Part 1)*
        ```text
        Repository: [имя_проекта]
        Files analyzed: [число]
        Estimated words: [число]

        Directory structure:
        └── [строго визуальное дерево каталогов]
        ```
    * *Блок 2: Код файла*
        ```text
        ================================================
        FILE: [относительный/путь/к/файлу.py]
        ================================================
        ```python
        [содержимое файла]
        ```
        ```
    * *Блок 3: Умный заголовок (для Part 2 и далее)*
        ```text
        [Repository: <имя_проекта> | Part <номер_части>]
        (Это продолжение исходного кода. Структуру директорий проекта см. в Части 1)
        ```

### 4. Изоляция шума (Noise Reduction Parameters)
Элементы, подлежащие безусловному отсеиванию на этапе фильтрации.

* **Исключения по системным директориям:** Строго игнорируются `.git`, `node_modules`, `venv`, `env`, `__pycache__`, `.pytest_cache`, `dist`, `build`, `.idea`, `.vscode`.
* **Исключения по Lock-файлам:** Строго игнорируются `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `poetry.lock`, `Gemfile.lock`.
* **Исключения по паттернам:** Все пути, совпадающие с правилами локального файла `.gitignore` (через библиотеку `pathspec`).
* **Исключения по объему:** Если размер файла `> --max-size` (в КБ), его содержимое не считывается. В документ записывается строка: `[СОДЕРЖИМОЕ ПРОПУЩЕНО: Размер файла превышает лимит --max-size]`.
* **Исключения по типу контента:** Если при чтении файла возникает `UnicodeDecodeError` (попытка прочитать бинарник как текст), файл маркируется как нечитаемый. В документ записывается: `[СОДЕРЖИМОЕ ПРОПУЩЕНО: Бинарный файл или неверная кодировка]`.
* **Подсчет слов:** Выполняется строго через применение функции `len(string.split())` к прочитанному чанку текста.

### 5. Обработка отказов (Fault Tolerance & Edge Cases)
Матрица ошибок с точными реакциями системы.

* **Ограничения среды:** Потребление оперативной памяти должно сохранять сложность O(1) относительно общего веса контента репозитория (список путей хранится в памяти, но контент читается потоково).
* **Матрица ошибок:** 
    * `[Триггер: Не указаны ни --url, ни --path]` -> `[Система выводит текст: "КРИТИЧЕСКАЯ ОШИБКА: Необходимо указать --url или --path"]` -> `[Fallback: Завершение работы с кодом 1]`.
    * `[Триггер: Указаны одновременно --url и --path]` -> `[Система выводит текст: "КРИТИЧЕСКАЯ ОШИБКА: Укажите только один источник (--url ИЛИ --path)"]` -> `[Fallback: Завершение работы с кодом 1]`.
    * `[Триггер: Указан --path, но директория не существует]` -> `[Система выводит текст: "КРИТИЧЕСКАЯ ОШИБКА: Директория не найдена"]` -> `[Fallback: Завершение работы с кодом 1]`.
    * `[Триггер: Отсутствует подключение к сети при клонировании --url]` -> `[Система выводит текст: "ОШИБКА СЕТИ: Невозможно клонировать репозиторий"]` -> `[Fallback: Завершение работы с кодом 1]`.

---

### 6. Ожидаемый артефакт (Final Deliverable)
Четкий критерий приемки (Definition of Done).

* **Результат:** Исполняемый скрипт на языке Python (`repo_to_doc.py`) и файл `requirements.txt`.
* **Тестовый прогон:** 
    * **Ввод:** `python repo_to_doc.py --path ./target_repo --output output_doc.txt --max-size 100 --split-words 1000`
    * **Ожидаемый визуальный вывод (CLI):**
        ```text
        ✅ Обход завершен!
        📂 Обработано файлов: [X]
        🚫 Пропущено (превышен лимит размера или бинарники): [Y]
        🧮 Общее количество слов: [Z]

        📄 Документация разбита на [N] файл(а):
        - output_doc_part1.txt (1000 слов)
        - output_doc_part2.txt (Z-1000 слов)
        ```
    * **Ожидаемый логический вывод (FS):** Наличие на диске файлов `output_doc_part1.txt` и `output_doc_part2.txt`, где начало второго файла содержит корректный контекстный "умный заголовок".
